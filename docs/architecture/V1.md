# Architecture proposée — V1

- **Statut** : APPROUVÉE, architecture de référence de la V1
- **Principe** : monolithe modulaire d'abord; extraction de services seulement quand une mesure le justifie.

## Stack

- **Web et API** : Next.js App Router + TypeScript strict.
- **UI** : composants accessibles, CSS Modules ou couches CSS, tokens sémantiques dans `src/styles/tokens.css`.
- **Langues** : `next-intl`, routes FR/EN et catalogues typés.
- **Données** : PostgreSQL Supabase, migrations versionnées.
- **Identité** : Supabase Auth avec sessions serveur.
- **Médias** : Supabase Storage, chemins privés par utilisateur et politiques RLS.
- **MCP** : SDK TypeScript officiel, endpoint HTTP distant dans un module serveur dédié.
- **Paiement** : Stripe Checkout, Customer Portal et webhooks signés.
- **Déploiement proposé** : Vercel pour l'application, Supabase managé pour les données.
- **Qualité** : ESLint, Prettier, Vitest, Testing Library, Playwright, GitHub Actions.

Les versions des dépendances seront épinglées dans le lockfile au démarrage du développement, après vérification de leur documentation actuelle.

## Modules

```text
src/
  app/                 # routes et composition
  components/          # composants partagés
  features/
    auth/
    categories/
    events/
    goals/
    schedules/
    billing/
    mcp-connections/
  server/
    auth/              # identité vérifiée
    domain/            # cas d'usage partagés web + MCP
    mcp/               # transport, outils et mapping d'erreurs
    repositories/      # accès PostgreSQL
    quotas/             # comptage et limites atomiques
  i18n/
  styles/tokens.css
```

Le web et le MCP appellent les mêmes cas d'usage dans `server/domain`. Les contrôleurs ne contiennent ni règle métier ni accès direct non contrôlé aux tables.

## Données principales

- `profiles(user_id, display_name, username, avatar_path, locale, timezone)`
- `categories(id, user_id, parent_id?, title, description, image_path, timestamps)` avec référence propriétaire récursive, unicité du titre parmi les catégories sœurs et interdiction des cycles.
- `events(id, user_id, category_id, title, description, timestamps)`
- `event_goals(event_id, goal_id, user_id, created_at)` relie plusieurs actions à plusieurs résultats sans traversée inter-utilisateur.
- `goals(id, user_id, category_id, title, description, status, goal_type, target_value, unit, period, timestamps)`
- `schedules(id, user_id, event_id, starts_at, ends_at?, status, recurrence, recurrence_interval, recurrence_weekdays, recurrence_ends_at?, recurrence_timezone, timestamps)` : seul un événement est planifiable.
- `schedule_occurrence_completions(schedule_id, occurrence_starts_at, user_id, completed_at)` valide une occurrence sans terminer toute la récurrence.
- `goal_check_ins(goal_id, period_start, user_id, value, completed_at)` conserve l’historique quotidien, hebdomadaire, mensuel ou unique.
- `media_assets` et tables de liaison pour plusieurs images d'événements/objectifs.
- `subscriptions(user_id, plan, status, provider_ids, period_dates)`
- `usage_periods(user_id, period, mcp_calls, storage_bytes)`

Chaque table métier contient `user_id`, des clés étrangères et RLS. Les politiques combinent rôle authentifié et propriété; une mise à jour contrôle l'ancienne et la nouvelle propriété. Les clés privilégiées ne quittent jamais le serveur.

Les catégories utilisent une liste d’adjacence (`parent_id`) et sont présentées comme un arbre calculé, sans colonne de profondeur figée. Un objectif décrit le résultat; un événement décrit l’action et peut avancer plusieurs objectifs; une planification place uniquement cet événement dans le temps. Les planifications récurrentes restent des règles uniques; des fonctions PostgreSQL `security invoker`, soumises à RLS, calculent les prochaines occurrences ou une plage interne bornée à 31 jours en respectant le fuseau et les changements d’heure.

## Authentification MCP

Le serveur MCP expose `/mcp` en Streamable HTTP stateless. La ressource protégée publie ses métadonnées RFC 9728 et délègue OAuth 2.1 à Supabase Auth : code d’autorisation avec PKCE, enregistrement dynamique, consentement explicite sur `/oauth/consent`, jetons JWT vérifiés par `getClaims` et révocation depuis le dashboard. La route de consentement est une route technique non localisée explicitement exclue du filtre 404. Les requêtes de données utilisent le jeton utilisateur et restent soumises aux RLS. Le rôle privilégié sert uniquement au compteur atomique.

Supabase OAuth Server est encore en bêta. Il émet des jetons de renouvellement mais ne prend actuellement en charge que les scopes standard `openid`, `email`, `profile` et `phone`; la compatibilité durable de renouvellement doit donc être prouvée sur chaque client annoncé avant publication.

## Quotas

- Réservation atomique d'un appel avant l'exécution d'un outil MCP.
- Remboursement seulement en cas d'erreur interne; les requêtes invalides/abusives restent comptées selon une règle documentée.
- Stockage calculé depuis les objets possédés, avec compteur réconciliable.
- Les webhooks Stripe sont idempotents; le serveur reste la source de vérité pour les droits.

## Risques à traiter tôt

- Compatibilité et autorisation différentes entre clients MCP.
- Contournement des quotas par concurrence.
- Fuite inter-utilisateurs via RLS ou chemins Storage.
- Divergence des règles entre dashboard et MCP.
- Coût/support des médias par rapport aux offres.
